# Security Self-Assessment

This document is an internal security self-assessment of the
`agent-auth`, `things-bridge`, and `things-cli` components as a
system. It follows the structure of the
[CNCF TAG-Security self-assessment
template](https://tag-security.cncf.io/community/assessments/guide/self-assessment/)
so that an outside reviewer (or a future version of the maintainer)
can pick up this document cold and understand the system's security
posture, trust boundaries, and residual risks in a single pass.

Content in this document is derived from `SECURITY.md`,
`design/DESIGN.md`, `design/SSDF.md`, `design/ASVS.md`, and the ADRs
under `design/decisions/`. Where a section summarises another
document, the link is given inline; the companion document is the
source of truth and this assessment rolls up on a cadence.

## 2. Overview

agent-auth is a token-based authorization system that gates AI-agent
access to macOS applications (initially Things 3, via AppleScript).
It issues scoped, HMAC-SHA256 signed tokens; requires just-in-time
human approval for `prompt`-tier operations; and exposes a token
lifecycle CLI and HTTP API.

### Background

Modern AI agents (Claude, Cursor, etc.) can drive local desktop
applications via scripting bridges (AppleScript on macOS). Without a
scoped authorization layer, an agent's capabilities default to the
user's full permissions on the host. agent-auth introduces a narrow,
auditable waist between the agent and the application:

- every agent call carries a bearer token whose scopes are verified
  on each request,
- sensitive operations (`prompt`-tier) are held pending a real-time
  human approval notification, and
- every authorisation decision is written to an append-only audit
  log.

### Actors

The system is composed of four cooperating processes, each in a
distinct trust zone:

- **agent-auth** — trust root. Holds the HMAC signing key and
  AES-256-GCM encryption key (both in the OS keyring), owns
  `tokens.db`, and is the sole issuer / validator of tokens. Exposes
  `/agent-auth/v1/...` over HTTP on `127.0.0.1`.
- **things-bridge** — bearer of tokens. Re-validates every incoming
  request with agent-auth before acting. Contains no Things 3
  logic; shells out to a Things-client CLI per request.
- **things-client-cli-applescript** — command-line `osascript`
  wrapper that runs on the macOS host and emits JSON on stdout.
  Trust assumption is that its invoker (things-bridge) is
  legitimate.
- **things-cli** — thin HTTP client for things-bridge, used by the
  human operator or an AI agent. Stores credentials in the system
  keyring.

Trust-boundary diagram and threat model are in
[`SECURITY.md` § Trust boundaries](../SECURITY.md#trust-boundaries)
and
[`SECURITY.md` § Threat model](../SECURITY.md#threat-model).

### Actions

The security-relevant actions each actor performs, and the checks
that gate them:

- **Token creation / modification / revocation / rotation** — the
  management endpoints on agent-auth require a bearer token in the
  reserved management family carrying `agent-auth:manage=allow`.
  The management token is bootstrapped on first server start into
  the OS keyring; see [ADR
  0014](decisions/0014-management-endpoint-auth.md).
- **Token validation** — `POST /agent-auth/v1/token/validate`
  verifies the HMAC signature, checks expiry, confirms
  non-revocation, and resolves the scope tier (`allow` / `prompt`
  / `deny`).
- **JIT approval** — `prompt`-tier operations are held server-side
  and a pluggable notification plugin asks the human operator for
  a decision. A resolved `deny` returns `401`; `allow` proceeds.
- **Refresh-token reuse detection** — presenting a previously used
  refresh token triggers family-wide revocation to bound the blast
  radius of a stolen token. See [ADR
  0011](decisions/0011-refresh-token-reuse-family-revocation.md).
- **Cross-component trust** — things-bridge never acts on a token
  presented by things-cli without re-validating with agent-auth;
  things-client-cli-applescript argv is constructed from
  schema-validated request parameters, never passed through
  verbatim.
- **Audit log writes** — every token-lifecycle and
  authorisation-decision event is written to agent-auth's
  append-only audit log before the response is sent. Token
  values are never logged; only family IDs.

### Goals

- **Constrain agent authority** — an AI agent's access to host
  applications is mediated by a scoped token, not the user's
  ambient permissions.
- **Human-in-the-loop for sensitive operations** — `prompt`-tier
  scopes cannot be self-approved by the agent; a human notification
  must be answered before the operation proceeds.
- **Auditable decisions** — every authorisation decision is
  reconstructable after the fact from the append-only audit log.
- **Least-privilege token families** — per-agent token families
  carry only the scopes they need; revocation of a family
  invalidates both access and refresh tokens atomically.
- **Supply-chain integrity** — every release ships with keyless
  cosign signatures ([ADR
  0016](decisions/0016-release-supply-chain.md)), SPDX SBOMs, and
  SLSA Build Level 3 provenance ([ADR
  0020](decisions/0020-slsa-build-provenance.md)).

### Non-goals

- **Multi-user / multi-tenant deployment.** agent-auth binds to
  `127.0.0.1` and assumes a single human operator. SC-8
  (transmission confidentiality) for devcontainer-to-host traffic
  is a known gap tracked in
  [#101](https://github.com/aidanns/agent-auth/issues/101).
- **Defence against an attacker with arbitrary code execution on
  the host.** A local-code-execution attacker can extract keys
  from the OS keyring (within policy), read the SQLite database,
  or impersonate the server.
- **Rate limiting or DoS resistance.** No in-product rate limiting
  today; mitigations would be reverse-proxy-level. Tracked in
  [#102](https://github.com/aidanns/agent-auth/issues/102).
- **Cryptographic audit-log integrity (hash-chaining).** The log
  is append-only but not tamper-evident. Tracked in
  [#103](https://github.com/aidanns/agent-auth/issues/103).
- **Hardened-runner requirements beyond GitHub-hosted.** Release
  artefacts are produced on GitHub-hosted runners; a compromised
  runner producing a consistently signed malicious bundle is
  accepted residual risk with Rekor as the detection signal.
- **Enterprise governance.** Personnel, supply-chain (upstream),
  physical, and organisational controls from NIST SP 800-53 are
  explicitly out of scope — see [`SECURITY.md` §
  Cybersecurity standard](../SECURITY.md#cybersecurity-standard).

## 3. Self-Assessment Use

This self-assessment is an **internal analysis** written by the sole
project maintainer, not an independent audit. It serves three
audiences:

- **Users and prospective users** who want to understand the
  system's security guarantees and residual risks before deploying
  agent-auth on their own machines.
- **Contributors and reviewers** who need a consolidated picture
  of the threat model, trust boundaries, and compliance posture
  before proposing changes that cross those boundaries.
- **External security reviewers** (including CNCF TAG-Security
  joint assessments, if agent-auth ever enters that path) who want
  a pre-structured starting point rather than having to reverse-
  engineer the project from its code and issue tracker.

Production deployments remain the user's responsibility. The
assessment's scope ends at the trust boundaries documented in
[`SECURITY.md`](../SECURITY.md).

## 4. Security Functions and Features

### Critical security components

Components whose compromise would defeat agent-auth's security
model:

- **HMAC signing key and AES-256-GCM encryption key** (in the OS
  keyring; see
  [ADR 0008](decisions/0008-system-keyring-for-key-material.md)).
  Compromise of either key allows token forgery or cleartext recovery
  of encrypted scope data.
- **`agent-auth serve` process.** Holds both keys in memory during
  its lifetime; is the sole token validator; runs the
  (in-process) notification plugin (see #6 for out-of-process
  migration). Compromise is equivalent to issuing arbitrary tokens.
- **`tokens.db` (SQLite).** Stores encrypted scopes, HMAC
  signatures, family linkage, and revocation timestamps. Direct
  modification is detectable via AES-GCM auth-tag failures but
  deletion (and therefore token revocation) is not cryptographically
  protected beyond append-only log writes.
- **Management token family.** Gates all lifecycle endpoints; its
  refresh token lives in the OS keyring and is bootstrapped on
  first server start. Loss of keyring access forces a recreate-on-
  restart path.
- **Release supply chain** (GitHub Actions workflows with
  `id-token: write`). A compromised release-publish workflow could
  produce a cosign-signed, SLSA-attested malicious bundle. Mitigated
  by SHA-pinning release-adjacent third-party actions
  ([ADR 0016](decisions/0016-release-supply-chain.md),
  [ADR 0020](decisions/0020-slsa-build-provenance.md), and the
  SHA-pin policy in
  [`.claude/instructions/tooling-and-ci.md`](../.claude/instructions/tooling-and-ci.md)).

### Security-relevant components

Components that enforce or support the security model but whose
compromise would not by itself defeat it:

- **`things-bridge serve` process.** Re-validates every incoming
  token with agent-auth; holds no cryptographic material. Compromise
  would allow an attacker to forward unauthorised Things 3 commands,
  but only within the scopes of tokens they could present.
- **`things-client-cli-applescript`.** Pure AppleScript wrapper
  emitting JSON; has no trust relationship with the caller's token.
- **Audit log.** Append-only today; cryptographic chaining is a
  follow-up (#103).
- **Cybersecurity, SDLC, and application-security standards** —
  NIST SP 800-53 ([`SECURITY.md` §
  Cybersecurity standard](../SECURITY.md#cybersecurity-standard)),
  NIST SSDF ([`design/SSDF.md`](SSDF.md)), and OWASP ASVS v5 L2
  ([`design/ASVS.md`](ASVS.md)).
- **QM/SIL posture.** Declared in
  [`design/ASSURANCE.md`](ASSURANCE.md).

## 6. Secure Development Practices

### Development pipeline

- **Conventional Commits + signed commits.** Every commit to
  `main` must be signed, enforced by a GitHub repository ruleset
  targeting the default branch with the `required_signatures` rule.
  Contributor instructions in
  [`CONTRIBUTING.md`](../CONTRIBUTING.md).
- **Gated CI checks.**
  - `check` / `test` (unit + integration; pytest + Docker-backed
    service integration),
  - `lint` / `format` (ruff, shellcheck, shfmt, mdformat, taplo,
    treefmt),
  - `typecheck` (mypy + pyright, both in strict mode with
    module-level ratcheting),
  - `pip-audit` (dependency vulnerability scan),
  - `reuse` (per-file SPDX licensing compliance),
  - `verify-standards` (checks the portable-standards invariants:
    Dependabot ecosystem coverage, task-runner coverage,
    required-file presence, `/v1/` route versioning, etc.),
  - `verify-design` and `verify-function-tests` (every leaf in the
    functional decomposition is allocated in the product breakdown
    and has test coverage),
  - `ripsecrets` pre-commit hook + CI step to block accidental
    secret commits.
- **Dependency management.** Weekly Dependabot PRs for `pip` and
  `github-actions`, with minor/patch updates grouped and majors
  landing individually for review.
- **Signed releases.** `release-publish.yml` on tag push attaches
  a keyless cosign signature bundle per artefact, an SPDX 2.3 SBOM
  per artefact, and a SLSA Build L3 `multiple.intoto.jsonl`
  provenance. Release-adjacent GitHub Actions are pinned to
  commit SHAs; read-only workflows stay on floating-major tags.
  See [ADR 0016](decisions/0016-release-supply-chain.md),
  [ADR 0020](decisions/0020-slsa-build-provenance.md).
- **Reviewer discipline.** Solo-maintainer project — reviews are
  self-reviews gated by CI. Releases are driven by semantic-release
  on merge to `main`; version derivation is reviewed at the PR
  commit-message stage (the PR *is* the review gate, since there is
  no downstream release PR). See
  [ADR 0026](decisions/0026-semantic-release-autorelease.md).
- **Container / image signing.** Not applicable; the project
  ships source distributions and wheels, not OCI images.
- **Vulnerability checks.** `pip-audit` runs daily on `main` and
  on every PR; Dependabot tracks action and package CVEs.
- **Fuzz / mutation testing.** Mutation testing is tracked as a
  standards-level follow-up; see
  [`.claude/instructions/testing-standards.md`](../.claude/instructions/testing-standards.md).

### Communication channels

- **Internal (maintainer ↔ AI collaborators).**
  `.claude/instructions/*.md` and `CLAUDE.md` document working
  agreements; `design/decisions/` ADRs record non-obvious choices.
- **Inbound (issues, support questions, PRs).** GitHub issue
  templates under [`.github/ISSUE_TEMPLATE/`](../.github/ISSUE_TEMPLATE)
  (bug, feature, security-redirect); `SUPPORT.md` points users
  at issues for non-security requests.
- **Outbound (release notes, advisories, changelog).**
  `CHANGELOG.md` is generated from Conventional Commit subjects and
  bodies by semantic-release on every release; the GitHub release
  body carries the same content. Older hand-written sections
  (pre-ADR 0026) remain in Keep-a-Changelog format. Security
  advisories go through
  [GitHub private vulnerability
  reporting](https://github.com/aidanns/agent-auth/security/advisories/new).

### Ecosystem

- **Depends on.** `setuptools-scm` (version derivation), Sigstore /
  cosign (keyless signing), Syft (SBOM), SLSA generator (provenance),
  REUSE (per-file SPDX), GitHub Actions, `uv` (lockfile + dev
  venv), go-task (task runner), mypy / pyright, ruff, Dependabot.
- **Used by.** Personal use on the maintainer's machine; no
  published downstream consumers. The README documents the `pipx`
  / `uv tool install` path for users who want to try it.
- **Integration points.** macOS keyring (via `keyring`), Things 3
  via AppleScript, OS notification surfaces through the
  configurable JIT plugin.

## 7. Security Issue Resolution

### Responsible disclosure process

See [`SECURITY.md` § Vulnerability
reporting](../SECURITY.md#vulnerability-reporting). Summary:

- **Do not** open a public GitHub issue for a suspected
  vulnerability.
- Use
  [GitHub private vulnerability reporting](https://github.com/aidanns/agent-auth/security/advisories/new)
  (the repo's `/security/advisories/new` endpoint).
- Expect an acknowledgement within one week given the project's
  best-effort support posture documented in
  [`SUPPORT.md`](../.github/SUPPORT.md).

### Vulnerability response process

Because agent-auth is a solo-maintained personal project, the
maintainer is both responsible for triage and remediation. The
working flow, derived from SSDF RV.\*:

1. **Acknowledgement.** Reply on the private advisory thread with
   a received-on date.
2. **Triage.** Reproduce on the current `main`; score impact and
   likelihood using NIST SP 800-30 Rev 1 (same rubric as the
   in-repo STRIDE tables). Escalate to a `High × Medium-or-above`
   finding if a working exploit in a documented threat surface
   is confirmed.
3. **Fix under embargo.** Develop the fix in a private worktree /
   private fork; CI gates and the standards-review checklist still
   apply.
4. **Coordinated release.** Cut a patch release via the usual
   semantic-release + release-publish workflow (merge a `fix:`
   commit to `main`), timing the advisory publication to the
   release. Include a `SECURITY.md` entry describing the class of
   issue (not the primitive, if the advisory embargo is still
   sensitive); the CHANGELOG bullet is derived from the commit
   subject.
5. **Post-mortem.** Open a public follow-up issue for any
   structural mitigation that could prevent the class of issue.

### Incident response

If a suspected compromise is reported (a leaked signing key, a
forged release, a malicious action version landing in main):

1. **Triage.** Confirm the compromise with concrete evidence
   (Rekor entry mismatch, unexpected git log, unauthorised tag).
2. **Containment.** Rotate the affected key material in the OS
   keyring; revoke the compromised token families; disable the
   affected GitHub Actions permission if the PAT or app token is
   suspected.
3. **Notification.** Publish a GitHub Security Advisory on the
   repo describing the scope of the compromise, the affected
   releases, and the verification steps users should run (`cosign verify-blob` and `slsa-verifier verify-artifact` against a
   known-good prior release).
4. **Remediation.** Yank or mark-as-compromised any affected
   release; cut a new release with rotated material; document the
   timeline in `SECURITY.md` or a dedicated advisory doc.
5. **Post-mortem.** Update this self-assessment, the threat model,
   and the relevant ADRs.

## 8. Appendix

### Known issues over time

No confirmed security issues have been reported or discovered since
the project's first commit (2026-04-19). The GitHub `#security`
issue label and
[Security Advisories tab](https://github.com/aidanns/agent-auth/security/advisories)
will index any future findings. Internal threat-model refreshes are
required by
[`.claude/instructions/plan-template.md`](../.claude/instructions/plan-template.md)
before any security-relevant change lands.

### OpenSSF Best Practices

- **OpenSSF Scorecard** — implemented as a CI gate in
  `.github/workflows/scorecard.yml` (weekly cron + push to `main`).
  SARIF is uploaded to the Security tab; results are published to
  `api.scorecard.dev` and rendered as a badge on the README. The
  workflow fails when the aggregate score drops below a ratcheting
  floor (`SCORECARD_MIN_SCORE`), covering branch protection, signed
  releases, dangerous workflows, token permissions, and dependency
  update cadence.
- **CII Best Practices badge** — not pursued today. A personal
  project with one maintainer would struggle to meet the
  "Contribution" and "Security team" criteria; revisit if the
  project grows contributors.
- **Sigstore / keyless cosign** — implemented for release
  artefacts ([ADR 0016](decisions/0016-release-supply-chain.md)).
- **SBOM on release** — implemented (SPDX 2.3, per artefact;
  [ADR 0016](decisions/0016-release-supply-chain.md)).
- **SLSA Build Level 3 provenance** — implemented
  ([ADR 0020](decisions/0020-slsa-build-provenance.md)).
- **REUSE 3.3 per-file licensing** — implemented; CI-gated
  ([ADR 0016](decisions/0016-release-supply-chain.md)).

### Case studies

The project is in personal use on the maintainer's development
machine; no external deployments are documented. Case studies will
be added here if the project is ever deployed beyond the
maintainer.

### Related projects and vendors

- **Anthropic Claude's tool-use HTTP gateway.** Solves an
  overlapping problem (authorising AI-agent tool calls) but at the
  model vendor's edge, not at the desktop app surface. agent-auth
  complements it by enforcing on the desktop host, where the
  actual AppleScript action runs.
- **OS-level Mandatory Access Control (macOS TCC, SELinux, AppArmor).**
  These constrain what the user — and any process running as that
  user — can do. agent-auth is narrower: it constrains what a
  specific AI agent's bearer token can do within the user's
  permission envelope, and adds the `prompt`-tier human-in-the-loop
  step that OS MAC layers do not provide.
- **OAuth 2.1 / RFC 6749.** Considered and explicitly not adopted;
  agent-auth uses a narrower custom token scheme documented in
  [ADR 0006](decisions/0006-token-format.md). OAuth's grant types
  and authorization server distinctions are overkill for a local,
  single-user project; re-evaluate if a remote-deployment scenario
  emerges.
