# Plan 0004 — Changelog scaffolding CLI (#297)

Sub-issue of #289. Phase 2 ergonomics on top of #295's hand-authored
YAML schema and #298's bot-mediated path. Adds a Python CLI that walks
contributors through composing one `changelog/@unreleased/*.yml`
entry per invocation, in either an interactive prompt-driven mode or
a one-shot non-interactive mode driven by flags.

## Scope

- `scripts/changelog/add.py` — type-annotated CLI. Public entrypoint
  invoked by `task changelog:add` (and the alias `task changelog-add`,
  matching the spelling in the issue body). Surfaces both an
  interactive walk-through (no flags) and a flag-driven non-interactive
  mode. Validation reuses #295's `version_logic` (`EntryType`,
  `validate_release_as`, `parse_entry_file`) and #295's lint helpers
  (`list_workspace_packages`, `list_present_changelog_files`) so there
  is exactly one source of truth for what counts as a valid entry.
- `scripts/changelog/wordlist.py` — small bundled wordlist (~150-300
  short, copy-pasteable English words) used to generate human-friendly
  filename slugs. No network call, no third-party dep — the wordlist is
  hand-curated from the EFF short list and kept in-repo with attribution
  in the module docstring.
- `Taskfile.yml` — add `changelog:add` (canonical) and `changelog-add`
  (alias matching the issue body) tasks that dispatch to the CLI. Both
  forward `{{.CLI_ARGS}}` so the non-interactive mode is callable as
  `task changelog:add -- --type fix --description "…" --pr 123`.
- `lefthook.yml` — add a non-blocking advisory pre-push hook that
  invokes `python scripts/changelog/add.py --check` and warns (to
  stderr) when the current branch has no `changelog/@unreleased/pr-<N>-*.yml`
  entry vs. `origin/main`. Advisory only: the CI lint
  (`changelog-lint.yml`) is the authoritative gate. The hook is fast
  (one `git diff --name-only` against `origin/main`, no network call).
- `scripts/changelog/tests/test_add.py` — pytest coverage of the
  non-interactive flag matrix, the wordlist slug generator, the
  collision retry, the `--release-as` validation reuse, the
  `--packages` workspace-membership check, the `--editor` flow, and
  the `--check` mode. Interactive prompts are tested via monkeypatched
  `input()` and dependency-injected I/O.
- `CONTRIBUTING.md` — extend the existing § "Authoring path" subsection
  to document the CLI mode #2 with both interactive and non-interactive
  examples; cross-reference #295's hand-authored path and #298's bot
  path so all three modes are linked.
- A bootstrap `changelog/@unreleased/pr-<N>-*.yml` for THIS PR
  (`type: feature`) generated by the new CLI itself (eat our own
  dogfood — the file's existence proves the CLI ran end-to-end).

## Out of scope

- ADR — this is a thin wrapper around #295's surface and #298's
  output format. No novel architectural decision.
- A `--push` mode that commits + pushes the YAML automatically; the
  CLI writes the file and exits, leaving the user to `git add` it
  alongside their feature commits.
- Per-package release-train inference (#275 territory).
- Refactoring the `version_logic` API; we consume it as-is.

## Key decisions

### CLI surface and prompt library

Stdlib `argparse` for flag parsing and stdlib `input()` / `print()` for
interactive prompts. No third-party prompt library. Rationale:

- The interactive flow is short — five questions max. A framework
  (`questionary`, `prompt-toolkit`, `rich`) buys arrow-key menus that
  are nicer but not load-bearing for a tool a contributor runs once
  per PR. The cost is a transitive-dep tree, slower import, and a new
  thing for `pip-audit` to keep up with.
- `version_logic.py` already uses pyyaml; that's the only external
  dep we need to round-trip a YAML on disk.

### TTY-detect fallback

`sys.stdin.isatty() and sys.stdout.isatty()` gates the interactive
walk-through. When run non-interactively (CI, piped, devcontainer
without a controlling TTY) and any required field is missing, the CLI
exits non-zero with a clear "this field is required in non-interactive
mode" message rather than blocking on `input()` forever.

### Random-words slug source

A ~200-word hand-curated list under `scripts/changelog/wordlist.py`
yields `200 * 200 = 40_000` two-word combinations — enough that
collisions inside one PR's filenames are vanishingly rare even after
rapid re-runs. Words are restricted to lowercase a-z (no apostrophes,
no digits, no hyphens) so the resulting `<word>-<word>` slug always
matches the lint's `[A-Za-z0-9_-]+` pattern. Source: a manual subset
of the EFF short wordlist (`https://www.eff.org/dice`) — public
domain, no attribution requirement, and curated for memorability.

The slug generator retries up to 5 times if the candidate filename
already exists; after 5 collisions it raises (a deterministic failure
mode is preferable to a silent overwrite).

### PR-number resolution

Three sources, in priority order:

1. `--pr <N>` flag (highest precedence — explicit always wins).
2. `gh pr view --json number --jq .number` against the current branch
   (best-effort; failures fall through silently).
3. Interactive prompt asking for the number.

`gh` is already a hard dependency of the repo (used by the changelog
bot, the merge bot, and the worked CLAUDE.md examples), so leaning on
it here is consistent.

### Description authoring

If `--description <text>` is passed, that text is used verbatim. If
the env var `$EDITOR` is set and `--description` is missing in
interactive mode, the CLI opens `$EDITOR` against a temp file
pre-populated with a one-line comment instructing the user. If
`$EDITOR` is unset and we're interactive, fall back to a single-line
`input()` prompt (with a documented limitation that newlines aren't
preserved).

### Validation order

Each invocation runs the following checks BEFORE writing any file —
fail fast with one error message per surfaced problem so the
contributor doesn't get a partially-written entry:

1. `--type` must be a known `EntryType` value.
2. `--packages` (if set) must all exist in
   `list_workspace_packages(repo_root)`.
3. PR number must be a positive integer (no leading zeros).
4. After composing the YAML in memory, run it through
   `parse_entry_file` (writing to a tmpfile then reading back) so the
   exact lint passes synchronously.
5. If `--release-as` is set, run `validate_release_as` against the
   *new entry plus the existing `@unreleased/` set* and the version
   from `detect_current_version`.

The CLI re-uses each helper directly from `scripts/changelog/lint.py`
and `scripts/changelog/version_logic.py`. No duplicate logic.

### `--check` mode for the lefthook hook

`--check` short-circuits the prompt flow and instead:

1. Resolves the current branch's diff against `origin/main`
   (`git diff --name-only origin/main...HEAD -- 'changelog/@unreleased/'`).
2. Prints a one-line warning to stderr if no entries were added /
   modified. Exits zero by default. With `--strict`, exits 1 on the
   same condition.

The hook's `lefthook.yml` entry uses `--check` (no `--strict`) so it
warns but doesn't block. Strict mode is a manual escape hatch for
contributors who *want* a hard local gate.

### Output format compatibility

The composed YAML matches the output of `bot.compose_yaml` byte-for-byte
when no extras (links, packages, release-as) are set, so the three
authoring paths converge on the same on-disk shape. Differences (extra
links, package list, release-as override) are appended in canonical
key order.

## Verification

- `task changelog:test` (extends the existing pytest target) to
  exercise `test_add.py`.
- `scripts/changelog/lint.py` parses the bootstrap entry that this PR
  adds (proven by the lint job in CI).
- `task lint`, `task format --check`, `task typecheck`,
  `task verify-standards`, `task verify-dependencies` all pass.
- `task changelog:add` invokes the CLI without ImportError or missing
  dep (proves the CLI is wired into the venv via `_bootstrap_venv.sh`).

## Standards review

- **Coding standards** — type-annotated, no raw tuples for structured
  data (use a small dataclass for the in-memory entry), procedures
  named with verbs (`compose_entry`, `derive_slug`, `prompt_for_type`).
- **Service design** — the CLI is a one-shot script; no service
  surface, no config file, no plugin system. It does NOT write to
  `$XDG_DATA_HOME` or `$XDG_STATE_HOME`; the only side-effect is the
  `changelog/@unreleased/pr-<N>-<slug>.yml` file in the repo.
- **Release and hygiene** — the bootstrap YAML THIS PR adds is
  `type: feature` and links to issue #297 + the PR.
- **Testing standards** — `test_add.py` exercises the CLI's public
  surface (argv + exit code, plus `compose_entry` directly); no
  internal-state assertions. Coverage: every branch in the validation
  cascade and the `--check` mode.
- **Tooling and CI** — `task changelog:test` already wraps
  `scripts/changelog/test.sh`; the new tests slot in without a
  separate runner. The new lefthook hook is purely local; no new CI
  job needed.

## Risks

- **Slug-list curation drift** — the wordlist could grow stale or
  acquire problematic terms. Mitigation: in-repo file with a one-line
  source attribution; review at the same cadence as any other static
  data file.
- **`gh` not installed** — the auto-detect step fails silently and the
  CLI prompts for the PR number. Documented in CONTRIBUTING.md.
- **`$EDITOR` opens a non-graphical editor in a sandboxed env** — the
  fallback to single-line `input()` covers the case. Documented in
  CONTRIBUTING.md.
